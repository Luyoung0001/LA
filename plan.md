### 1.3 详细设计说明

#### BTB (分支目标缓冲器)

| 参数 | 值 |
|------|-----|
| 容量 | 256项 × 2路组关联 |
| 索引方式 | PC[9:2] 作为索引，PC[31:10] 作为 Tag |
| 每项内容 | {Valid, Tag[21:0], TargetAddr[31:0], Type[2:0]} |
| 替换策略 | 伪LRU (2路需要1位跟踪) |
| 命中延迟 | 1周期 |
| 更新时机 | EX阶段确认分支实际方向和目标地址后回写 |

**工作流程：**
1. 用当前PC索引BTB
2. 比较Tag，命中且Valid则输出预测目标地址
3. 若BTB未命中但Gshare预测跳转，使用PC+Offset作为目标
4. 分支在EX阶段解析后，更新BTB（分配新项或更新现有项）

#### Gshare方向预测器

| 参数 | 值 |
|------|-----|
| 全局历史寄存器(GHR) | 12位（记录最近12条分支的方向） |
| 模式历史表(PHT) | 4096项 × 2位饱和计数器 |
| 索引计算 | GHR[11:0] XOR PC[11:0] |
| 更新 | 分支执行后，更新GHR和对应PHT项 |

**饱和计数器状态机：**
```
        ┌──────┐    实际不跳转    ┌──────┐
        │ 强不 │◄───────────────│ 弱不 │
        │ 跳转 │               │ 跳转 │
        │  00  │───────────────►│  01  │
        └──────┘    实际不跳转    └──────┘
                                     │        实际跳转
                                     │
        ┌──────┐    实际跳转       ┌──────┐
        │ 强跳 │◄───────────────│ 弱跳 │
        │ 转   │               │ 转   │
        │  11  │───────────────►│  10  │
        └──────┘    实际跳转       └──────┘
```
- 00 = Strongly Not Taken
- 01 = Weakly Not Taken
- 10 = Weakly Taken
- 11 = Strongly Taken
- 预测规则：若计数器高位(bit[1])为1则预测跳转

#### RAS (返回地址栈)

| 参数 | 值 |
|------|-----|
| 深度 | 16项 |
| 栈指针 | 4位栈顶指针 |
| Push时机 | D1阶段识别为call指令时 |
| Pop时机 | F1阶段识别为ret指令时（预测性弹出） |
| 恢复机制 | 分支预测错误冲刷时，需恢复RAS指针。维护检查点快照 |

**溢出处理：**
- Push时栈满：覆盖最旧项（环形栈实现）
- Pop时栈空：保持栈顶不变，使用默认预测（如BTB）

#### 预测更新与恢复

```
预测更新流水线：

F1: 使用预测结果取指
 ↓
D1: 识别指令类型，RAS操作
 ↓
EX: 实际计算分支方向和目标
 ↓
反馈: 发送 {预测正确/错误, 实际目标, 实际方向} 回BPU
 ↓
恢复: 若预测错误 → 冲刷流水线 → 从正确PC重新取指
     同时更新BTB/Gshare/RAS
```

### 2.3 详细设计说明

#### ITCM (指令紧耦合存储器)

| 参数 | 值 |
|------|-----|
| 容量 | 可配置：8KB / 16KB / 32KB / 64KB |
| 访问延迟 | 固定1周期 |
| 地址映射 | 通过基址寄存器配置，如 0x0000_0000 起始 |
| 特性 | 确定性延迟，无缓存未命中，通过写操作（调试）更新 |
| 使能控制 | 可通过CSR (控制状态寄存器) 动态开启/关闭 |

**优势场景：**
- 中断服务程序 (ISR) - 确定性延迟
- 实时控制循环
- 启动代码

#### L1 I-Cache

| 参数 | 值 |
|------|-----|
| 容量 | 16KB (可配置8KB/16KB/32KB) |
| 组织 | 2路组关联，缓存行 64字节 |
| 索引 | 128组，索引位 = addr[12:6] |
| Tag | addr[31:13] = 19位 |
| 替换 | 伪LRU每路1位 |
| 命中延迟 | 1周期 |
| 未命中处理 | 请求L2 Cache，可缓冲2个未命中请求 |

**未命中流程：**
1. 未命中 → 冻结Fetch Buffer写入
2. 发出64B行填充请求给L2 Cache
3. 关键16B数据到达 → 立即释放给Fetch Buffer（关键字优先）
4. 剩余数据顺序填充
5. 释放冻结

#### Fetch Buffer 设计

| 参数 | 值 |
|------|-----|
| 深度 | 4项（覆盖4周期取指延迟） |
| 每项宽度 | 128位 = 2条 × 32位指令 + 32位地址 + 有效位 |
| 输出能力 | 每周期最多输出2条完整指令（地址对齐） |
| 满条件 | 4项全有效 → 反压F1阶段 |
| 空条件 | 0项有效 → 暂停D1阶段 |

**状态机：**
```
   IDLE ──► FILLING ──► ACTIVE
    │         │           │
    │         │           │
    └── 清空 ─┴── 回退 ───┘
```

**设计要点：**
- 解耦取指速度和译码速度（I-Cache未命中时，Fetch Buffer可继续提供指令）
- 支持指令对齐：在下一条指令地址非对齐时，正确处理跨缓存行边界
- 分支预测错误冲刷时，同步清空Fetch Buffer

### 3.3 详细设计说明

#### 双发射约束条件

| 发射模式 | 条件 |
|----------|------|
| 双发射 (2条) | Inst0可发射 且 Inst1可发射 且 无资源冲突 且 Inst1不依赖Inst0 |
| 单发射 (仅Inst0) | Inst0可发射 但 Inst1不满足条件 |
| 阻塞 (0条) | Inst0不可发射（等待数据或资源） |

**资源限制（不能同时发射的组合）：**

| Inst0 | Inst1 | 是否允许 |
|-------|-------|----------|
| ALU操作 | ALU操作 | ✅ 允许 |
| ALU操作 | 乘/除 | ❌ (MDU只有1个) |
| 乘/除 | ALU操作 | ❌ (MDU只有1个) |
| Load | Load | ✅ (若L1D支持双端口或分Bank) |
| Load | Store | ✅ (Store进入Buffer) |
| Store | Load | ✅ (需正确旁路) |
| 分支 | ALU | ✅ |

#### 数据旁路详细设计

**旁路来源（按优先级排序，最新结果最高优先）：**

| 优先级 | 来源 | 数据阶段 | 延迟 |
|--------|------|----------|------|
| 1 (最高) | EX阶段ALU0结果 | 当前周期结束可用 | 0周期（EX→EX） |
| 2 | EX阶段ALU1结果 | 当前周期结束可用 | 0周期（EX→EX） |
| 3 | M1阶段Load数据 | 前1周期 | 1周期（M1→EX） |
| 4 | M1阶段Store Buffer转发 | 前1周期 | 1周期 |
| 5 | WB阶段写回数据 | 前2周期 | 2周期（WB→EX） |
| 6 (最低) | 寄存器堆 | - | 寄存器读取 |

**旁路匹配逻辑（每源操作数独立）：**
```
if (src_reg == ex0_rd && ex0_wen)     data = ex0_result;
else if (src_reg == ex1_rd && ex1_wen) data = ex1_result;
else if (src_reg == m1_rd && m1_wen)   data = m1_result;
else if (src_reg == wb_rd && wb_wen)   data = wb_result;
else                                    data = gpr_read_data;
```

#### 发射计分板

维护一个简洁的计分板跟踪未完成写回的寄存器：

```
┌──────────────────────────────────────────────────┐
│               GPR 计分板 (32位)                   │
│   bit[i] = 1 → 寄存器 Ri 有未完成的写操作        │
│                                                   │
│   发射时：检查 (scoreboard[rs1] == 1) ?           │
│           需旁路若源来自刚发射指令（同周期）       │
│           否则：等待写回（若不在旁路范围内）       │
│                                                   │
│   发射时设置：scoreboard[rd] = 1                  │
│   写回时清除：scoreboard[rd] = 0                  │
└──────────────────────────────────────────────────┘
```

#### MDU - 乘法器设计

**Booth-2 乘法器结构：**
```
A[31:0] × B[31:0]

  │   ...
  └── B[31:29]       ──► Booth选择器15 ──► 部分积15

```

**流水线寄存器插入（可选）：**
- 1周期模式：纯组合，需满足时序
- 3周期模式：Booth编码→Wallace树→加法器 各1级寄存器

| 分配 | Store指令进入EX阶段时分配（按序） |
| 提交 | Store指令到达WB阶段时标记为Committed |
| 释放 | Store数据写入L1D后释放 |

**状态转换：**
```
EMPTY → ALLOCATED (EX阶段写地址和数据)
      → COMMITTED (WB阶段确认提交)
      → MERGED (与后续Store合并)
      → WRITTEN (数据写入L1D)
      → RELEASED (释放回EMPTY)
```

**Store合并策略：**
- 若新Store地址与Buffer中某一未提交Store地址完全重叠或相邻
- 且Store大小可合并为word/halfword/byte操作
- 则覆盖或扩展该项数据，不分配新项

#### Store-Load 旁路逻辑

```
优先级编码旁路逻辑:

Load地址 addr_L, 大小 size_L

for 每个 Store Buffer 项 i (从最新到最旧):
    if (valid[i] &&
        addr_L在[addr_i, addr_i + size_i)范围内 &&
        Store_i比Load旧):
        提取重叠字节 → 转发给Load
        return

// 无匹配
从L1 D-Cache读取
```

#### L1 D-Cache 参数

| 参数 | 值 |
|------|-----|
| 容量 | 16KB (可配置8KB/16KB/32KB) |
| 组织 | 2路组关联 |
| 缓存行 | 64字节 |
| 索引位 | addr[11:6] = 6位 → 64组 |
| Tag | addr[31:12] = 20位 |
| 状态位 | V(alid), D(irty) 每缓存行 |
| 写策略 | 写回 (Write-Back) |
| 分配策略 | 写分配 (Write-Allocate) |
| 替换 | 伪LRU (2路仅需1位) |

**未命中处理：**
1. 分配Miss Status Holding Register (MSHR)
2. 发送64B行填充请求到L2 Cache
3. 若为Load未命中：关键16B数据到达即唤醒流水线
4. 若为Store未命中：先分配缓存行，再写入Store数据
5. 若需要替换脏行：先写回脏行到L2

### 6.3 详细设计说明

#### 写回顺序保证

顺序双发射处理器中，写回必须按程序顺序：

```
保证机制：
1. 流水线本身是顺序的 → EX阶段结果按序到达WB
2. 若Inst0和Inst1同时到达WB：
   - Inst0写入GPR端口0
   - Inst1写入GPR端口1
3. 若Inst0延迟(如Load未命中)，Inst1需等待：
   - Inst1在EX完成后停留在流水线寄存器
   - 直到Inst0完成WB，Inst1才能进行WB
```

#### 写回冲突处理

当同周期写回的两条指令具有相同目的寄存器：

```
规则：后发射的指令（Inst1）结果优先

if (rd0 == rd1 && wen0 && wen1):
    写端口0写入 data0  // 被覆盖也无妨
    写端口1写入 data1  // 最终正确值
    # 或者在写前仲裁：
    # 如果rd0==rd1，wen0 = 0，只让wen1=1
```

#### 旁路网络最终级

WB阶段是旁路网络的最后一站，数据在这里可用性确认后广播给所有等待的指令。

## 附录：关键时序分析

### 最可能的关键路径

| 路径 | 延迟组成 | 预估延迟 |
|------|----------|----------|
| F1→F2 | BPU选择 + ITCM/I$ SRAM读取 + Fetch Buffer写入 | ~1.2ns |
| D1 | 译码 + 依赖分析 + 旁路MUX选择 + 寄存器读取 | ~0.9ns |
| EX(ALU) | 操作数锁存 + ALU运算 + 结果锁存 | ~0.8ns |
| M1 | 地址→TLB→Tag比较→数据读取→对齐 | ~1.5ns |
| WB | 写端口仲裁 + GPR写入 | ~0.6ns |

### 时钟周期建议

- **目标频率：** 800MHz - 1.2GHz (28nm)
- **最坏路径：** M1 L1D访问或MDU除法
- **优化方向：** 若M1成为瓶颈，可增加M1→M2访存拆分
